# Mission 017 - Golden Run Telemetry Truth Hardening

Goal:
Use the next Golden Run as a recursive self-improvement hardening run to turn
the `/golden-run` board into an SSOT-ready operator surface.

## Mission objective

Mission 017 is Part 1 of a two-part Golden Run sequence.

This packet must remediate the telemetry truth seams revealed by the
2026-04-08 audit by using a governed B.R.R.R loop:

1. `Build` one bounded Golden Run telemetry truth seam
2. `Review` it against live workforce / flow / build evidence
3. `Refine` the source-precedence or provenance contract
4. `Repeat` until the board can honestly act as a Golden Run SSOT

Mission 017 succeeds only if the board becomes able to truthfully answer:

- what the active Golden Run actually is
- what phase it is actually in
- which touch points are live, derived, or stale
- what the operator should trust first

without collapsing multiple unrelated ladders into one fake clean runway.

## Why now

The board is back up, but the audit showed it is still not trustworthy enough
to anchor the next hardening cycle.

The strongest current risks are:

- hero focus can drift to the wrong mission
- mission / proof / publish lanes can read `idle` while live work exists
- `live` badges can overstate freshness
- the one-button systems check can pass while semantics are still wrong

That means the Golden Run page itself is now the next bounded reliability seam.

## Recursive loop method

Every Mission 017 loop must follow B.R.R.R:

### Build

- choose one telemetry seam only
- name the source-precedence hypothesis before mutation
- patch the smallest authoritative source-precedence or provenance contract
- do not widen into styling or unrelated card work

### Review

- compare `/api/flow`, `/api/build`, `/api/golden-run/test`, and live workforce
  truth for the same mission
- label each compared touchpoint as `live`, `derived`, `stale`, or `unknown`
- collect exact mismatches
- require operator-readable receipts

### Refine

- tighten the contract
- reduce ambiguity
- patch the next smallest truth break exposed by review

### Repeat

- admit the next loop only after:
  - Codex verdict
  - Voltaris V2 verdict
  - mission lead verdict
- require that vote before every refine / repeat cycle
- if those do not align, narrow scope before continuing

## RSI stop conditions

Stop the loop and return a blocker if:

- the board still cannot resolve one governing mission object
- provenance/freshness cannot be stated honestly
- the test surface still self-certifies without independent truth
- the pass starts redesigning surfaces or adding new telemetry features instead
  of hardening truth
- live / derived / stale semantics remain ambiguous after review
- the packet starts widening into redesign or adjacent product work
- diminishing returns become obvious before SSOT readiness is achieved

## Acceptance criteria

1. The Golden Run hero resolves from the active admitted mission first.
2. Mission / proof / publish / closeout reflect active mission truth rather than
   unrelated hero drift.
3. Each touchpoint can honestly report provenance/freshness.
4. The Golden Run systems check can fail when semantics are wrong.
5. One operator-visible proof bundle shows the board agrees with underlying
   live mission state.

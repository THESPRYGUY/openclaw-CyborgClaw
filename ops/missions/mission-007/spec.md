# Mission 007 — Full Flow Golden Run

Goal:
Run one governed end-to-end CyborgClaw golden run from idea drop through development publishing and closeout, then use a 10-loop recursive improvement cadence to harden the full operator path until it behaves like a trustworthy MVP flow under live use.

## Mission objective

Prove that CyborgClaw can move one bounded initiative through the full stack:

1. idea drop
2. shaping
3. PRD and dev-staging handoff
4. specialist seat design
5. strike-team composition
6. STUD preflight
7. mission execution
8. proof, review, and receipts
9. development publishing
10. closeout and archive posture

The mission is successful only if every stage remains truthful, reviewable, and bounded, with no hidden runtime authority and no missing paperwork.

## North star

This is not a surface-design mission.

This is a machine-readiness mission.

We are validating that CyborgClaw can:

- accept a real idea drop without false promotion
- produce a bounded PRD and dev-pack handoff
- author and publish a specialist seat without implying live runtime mutation
- compose and preflight a governed strike team through STUD
- execute a bounded mission while keeping telemetry and seat posture truthful
- collect proof, receipts, and review notes without losing the audit trail
- publish and close out the development outcome with a complete paperwork bundle

## Non-negotiable rules

- Runtime authority stays downstream. Publish never equals live.
- RSI remains draft-producing and eval-first only.
- STUD must block launch on true blockers.
- If telemetry is stale, partial, or projected, the UI must say so explicitly.
- One canonical agent baseball-card contract must hold across the main surfaces.
- Every major stage must emit or update a receipt path.
- Honest yellow is better than fake green.

## Golden-run stages

### Stage 1 — Idea capture and shaping

Ground truth lives in the lifecycle vocabulary:

- `idea`
- `exploring`
- `validated`
- `promoted`

The operator must always know whether the work is merely captured, actively shaped, or truly ready for promotion.

### Stage 2 — PRD and dev-staging handoff

The stage-gate truth must stay distinct:

- `draft`
- `in_hivemind`
- `prd_ready`
- `dev_pack_ready`
- `queued`

No stage may silently imply execution authorization unless the downstream queue, dev-pack, and receipt posture are actually present.

### Stage 3 — Agent seat forge

Agent Design Studio must stay seat-first and bounded:

- one specialist seat
- one primary specialty
- explicit doctrine packs
- explicit eval gates
- explicit publish/runtime separation

### Stage 4 — Strike-team hangar

Strike Team Design Studio must stay formation-first:

- commander
- core seats
- bench seats
- coverage gaps
- team blockers

It must never drift into hidden seat authoring.

### Stage 5 — STUD preflight and mission launch

STUD must prove:

- roster lock
- identity and kinship legality
- comms proof
- seat capability
- controlled sound-off
- paperwork path

### Stage 6 — Mission execution and truth rails

The mission room, inspectors, and roster wall must remain aligned about:

- live versus stale seats
- admitted versus projected posture
- current task and comms posture
- dispatch and review state

### Stage 7 — Proof, review, and closeout

The golden run must produce:

- dispatch ledger rows
- proof back-briefs
- review decisions
- artifact intake truth
- closeout packet

## The 10 loops

### Loop 1 — Idea drop truth

- Target seam: idea capture and lifecycle state
- Goal: prevent captured work from feeling promoted too early
- Main question: can the operator tell whether the idea is merely captured, being shaped, or truly ready?

### Loop 2 — Shaping and salon truth

- Target seam: shaping support, salon support, and planning outputs
- Goal: keep shaping discussion separate from planning truth
- Main question: can the operator tell what is discussion, what is planning, and what is approved next?

### Loop 3 — PRD and dev-staging truth

- Target seam: PRD-ready, dev-pack-ready, and execution authorization
- Goal: keep handoff stages distinct and receipt-backed
- Main question: can the operator explain why the work is PRD-ready versus queued?

### Loop 4 — Agent seat forge truth

- Target seam: Agent Design Studio
- Goal: keep seat readiness honest and next actions obvious
- Main question: can the operator design one seat without reading the whole machine?

### Loop 5 — Strike-team hangar truth

- Target seam: Strike Team Design Studio
- Goal: keep formation editing separate from seat authoring
- Main question: does the operator feel like they are composing a team, not editing hidden seat internals?

### Loop 6 — STUD boot truth

- Target seam: preflight and launch gating
- Goal: block false-green launches and surface exact blockers
- Main question: does STUD explain why launch is safe, risky, or blocked?

### Loop 7 — Mission execution truth

- Target seam: workforce mission room and live seat telemetry
- Goal: keep live, stale, degraded, and waiting states truthful
- Main question: when one seat drifts, does the system tell the operator calmly and accurately?

### Loop 8 — Proof and paperwork truth

- Target seam: docs rail, proof index, dispatch ledger, artifact intake
- Goal: ensure nothing important falls on the floor
- Main question: can the operator trace the mission evidence without hunting?

### Loop 9 — Development publishing and closeout truth

- Target seam: artifact workflow, publish rails, closeout packet, archive readiness
- Goal: make development publishing auditable and bounded
- Main question: can the operator tell what is draft, in review, final, and archived?

### Loop 10 — Full end-to-end golden run

- Target seam: the whole operator path
- Goal: prove that one fresh idea can move from capture through closeout
- Main question: can one operator complete the whole run without confusion about what is live, what is staged, and what happens next?

## Required evidence

The golden run must produce or update:

- idea record and lifecycle evidence
- PRD / dev-pack evidence
- seat publish bundle
- team publish bundle
- STUD preflight receipt
- mission execution ledger
- proof back-briefs and reviews
- truth/artifact intake receipts
- development publishing receipts
- final closeout bundle

## Success bar

Call Mission 007 successful only if:

- one fresh idea can traverse the full path
- lifecycle truth stays consistent across the surfaces
- seat and team design remain clearly separate
- publish and runtime-effective truth remain clearly separate
- STUD blocks false-green launches
- paperwork rails and receipts remain cross-linked enough to trust
- the operator always has one clear next action

## Stop conditions

Stop the mission immediately if:

- any surface implies publish equals live
- runtime authority drifts into Studio
- telemetry cannot be trusted and no stale/partial label is possible
- the paperwork path is broken badly enough that closeout would be unauditable
- the mission room or truth rails become misleading under live use
